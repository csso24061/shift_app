# シフト管理アプリ
シフト管理Webアプリ（MVP版）要件定義 ＆ 基本設計書特定の業界に依存せず、多様な勤務形態に対応できる汎用的なシフト管理・給与計算Webアプリケーションです。本ドキュメントは、最小限の実用性を持つ製品（MVP: Minimum Viable Product）の開発に向けた要件定義と、4種類の設計図（基本設計）を網羅しています。1. 要件定義（Requirements Definition）1.1 目的汎用的なシフト・給与の一元管理： 業界を特定せず、30分刻みの柔軟な勤務時間に対応できるシフト管理・給与計算システム。管理業務の効率化と提出漏れ防止： シフトの提出、調整、急な変更申請の処理、労基法に準拠した給与計算までのワークフローを自動化・効率化。厳格なプライバシー保護の提供： 個人情報である給与や労働時間データをシステム側で厳重に保護し、店舗全体のマネジメントと個人のプライバシーを両立。1.2 利用者の入出力（ユースケースベース）利用者（ロール）入力（Input）出力・返ってくるもの（Output）全体運営者・新規導入店舗の登録情報・店舗管理者アカウントの初期発行・登録済み店舗の一覧・各店舗のアカウント有効/無効状態店舗管理者（店長・社員等）・スタッフ全員のシフト調整・確定操作・スタッフの給与・時給設定・シフト提出締め日、変更期限、給与締め日の設定・スタッフからの変更申請への承認/却下操作・店舗全体の確定シフト表（カレンダー）・全スタッフの労働時間・給与一覧・シフト未提出者への催促バナー表示・スタッフからの変更申請の通知一般スタッフ（アルバイト等）・ログイン情報（ID/パスワード）・月ごとの希望シフト（30分刻み）・確定シフトに対する変更申請（当日1週間前まで）・自分および他スタッフの確定シフト表（他人の給与は見えない）・自身の当月/過去の労働時間と給与明細データ・未提出時のシフト提出催促ポップアップ1.3 制約事項技術プラットフォーム： スマートフォンおよびPCのブラウザから利用できる「Webアプリ」として構築する（ネイティブアプリは開発しない）。時間単位の制約： シフトパターンを固定せず、「30分刻み」で自由な時間（開始・終了時間）を入力・調整できる仕様とする。スケジュールの動的変更： シフトの提出締め日や給与締め日は固定せず、各店舗の管理者がシステム上で自由に入力・コントロール・変更できるようにする。セキュリティ制約： 給与・労働時間データは、閲覧権限を持つ本人および店舗管理者以外には絶対に開示されない構造とする。1.4 受け入れ基準（完成定義）3階層の権限管理： 「全体運営者」「店舗管理者」「一般スタッフ」がそれぞれ正しい認証情報でログインでき、権限に応じた画面メニューが正しく制御されていること。店長・社員の1ID兼任： 店舗管理者（店長・社員など）が、別アカウントに切り替えることなく、1つのログイン状態で「自身の希望シフト提出・給与閲覧」と「店舗全体のシフト作成・管理」をシームレスに行えること。30分刻みのシフト確定と公開： スタッフが30分単位で希望を提出でき、管理者がそれを調整して「確定・公開」した瞬間に、全員のカレンダーに反映されること。期限付き変更申請と管理者通知： シフト当日の1週間前まではスタッフ画面に変更申請ボタンが表示され、申請時に管理者画面へ通知が行き、管理者が承認するとシフトが自動書き換えされること。一律ルールの給与自動計算： 確定したシフトの労働時間から、一般的な労働基準法（実働8時間超の残業割増、22時以降の深夜手当など）に基づき、各スタッフの給与額が自動かつ正確に計算・表示されること。1.5 非目標（今回は作らない範囲）リアルタイムの出退勤打刻： タイムカード、QRコード、GPSなどによるリアルタイムの「打刻機能」は今回実装しない。スタッフ間のチャット・直接交換： アプリ内チャット機能や、スタッフ同士が直接シフトを交渉・交換する機能は含めない。複数店舗の掛け持ち（ヘルプ対応）： 複雑なデータ構造は対象外とし、「1スタッフ＝1店舗所属」に限定する。期限を過ぎたシフト変更のシステム処理： 当日の1週間前を過ぎた変更申請は、アプリ内での申請操作自体を不可とする。やり取りは外部（LINEや口頭など）で行い、最終結果のみを管理者がアプリ上で手動修正する運用とする。2. 機能・非機能要求（Requirements Backlog）2.1 機能要求リスト【高】（初期必須）：ログイン・ログアウト認証、ロールベース画面制御、希望シフト提出、シフト調整・確定、確定シフトカレンダー閲覧、期限付きシフト変更申請、一律ルールの給与自動計算、全体運営者の店舗管理。【中】（中期展開）：提出自動催促ポップアップ、変更申請の通知＆承認、期限コントロール設定、給与締め日設定、個人用給与確認画面、店舗用給与一覧画面、店舗スタッフアカウント管理。【低】（運用カバー）：管理者によるシフト直接修正（1週間前を過ぎた変更の手動書き換え）。2.2 非機能要求セキュリティ： 給与・労働時間データのデータベースクエリレベルでのアクセス制御、常時HTTPS、パスワードのハッシュ化（BCrypt推奨）。ユーザビリティ： レスポンシブ対応（スマートフォンUIの最適化）、直感的な30分刻みのカレンダー入力インターフェース、店長アカウントの1タップ権限・表示切り替え。性能・拡張性： レスポンスタイム3秒以内、締め日前の同時アクセスに耐える負荷耐性、各店舗のデータが混ざらない「マルチテナント」を意識した物理・論理分離設計。保守性： PHP (Laravel) や Ruby (Ruby on Rails) などのオールインワンフレームワークによる単一リポジトリ管理、給与計算ロジックの定数（設定値）切り離し管理。
